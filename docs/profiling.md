# CPU Profiling and Throughput Analysis

This document describes how to run the local-cluster stress harness to measure
throughput and generate CPU flame graphs for the HOPR transport pipeline.

## What it profiles

The harness spins up 2–9 HOPR nodes on the local machine against the **mock
chain** (`FullStateEmulator`), opens a full directed channel mesh, and drives
bulk 1-hop traffic through random `src → relay → dst` session triples.

The pipeline that appears in the flame graph includes:

- SPHINX packet encoding (`crypto/packet`)
- Proof-of-Relay ticket creation and validation
- Session segmentation (`transport/session`)
- Mixer delay scheduling (`transport/mixer`)
- Tag allocation and deduplication
- Relay-node SPHINX decryption and re-encryption

**Measurement model**: the harness measures **write throughput** (bytes sent into the
forward pipeline, src→relay→dst). Reading the echo back requires pre-allocated SURBs
on the exit node, which default session configuration does not supply. The forward path
alone is sufficient to saturate and profile all SPHINX/ticket/mixer hot paths.

**Out of scope**: the blokli/RPC chain connector does not run in this mode.
If the transport pipeline looks clean, that narrows the investigation to the
chain connector layer.

## The `[profile.profiling]` cargo profile

A dedicated cargo profile keeps release-level optimisations while preserving
full DWARF debug symbols:

```toml
[profile.profiling]
inherits = "release"
debug = 2
strip = false
```

This mirrors `[profile.memprof]` (used for allocation profiling) and is
required for `cargo flamegraph`/`perf` to resolve function names.

## Running the stress harness

### Basic throughput read (any platform)

No profiling dependency needed. Prints a per-window MB/s table and a summary.

```bash
cargo run --features testing \
    --example stress_cluster \
    -- --nodes 5 --mb 100
```

Example output:

```
     time(s)   window MB/s     cumulative
  ──────────────────────────────────────
        0.50         8.412        4.21 MB
        1.00        10.231       13.33 MB
        1.50        11.047       18.85 MB
        ...
  ──────────────────────────────────────
  Total: 100 MB in 9.42s — mean 10.61 MB/s
```

A stable or climbing series is the healthy baseline. A series that steadily
falls indicates backpressure or queue growth somewhere in the pipeline.

### In-process flame graph via pprof (Linux preferred; macOS requires frame-pointer build)

Build with the `profiling` feature to enable in-process CPU sampling. The
profiler starts **after** cluster bootstrap and channel setup so the flame
graph captures only the sustained workload.

```bash
cargo run --profile profiling \
    --features testing,profiling \
    --example stress_cluster \
    -- --nodes 5 --mb 100 --out flame.svg
```

Open the result:

```bash
open flame.svg          # macOS
xdg-open flame.svg      # Linux
```

The flame graph is generated by the [`pprof`](https://crates.io/crates/pprof)
crate using `SIGPROF`-based sampling at 1000 Hz (frame-pointer unwinding).
It captures CPU-active threads only — async tasks waiting for I/O do not appear.

**macOS caveat**: pprof's SIGPROF handler can conflict with tokio's multi-thread
runtime on Apple Silicon (exit code 133, `Trace/BPT trap`). Building with the
`frame-pointer` pprof feature (already set in `hopr-lib/Cargo.toml`) uses frame-pointer
unwinding instead of libunwind, which is async-signal-safe. On Apple Silicon the ARM ABI
mandates frame pointers so they are always available. If crashes persist, use `sample` or
Instruments instead (see the macOS section below).

### macOS native profiling with `sample`

If pprof is unstable on your machine, use the built-in `sample` command — no Xcode or
sudo required. Run the binary first, then immediately attach `sample` in a second terminal:

```bash
# Terminal 1 — start the stress run:
cargo build --profile profiling --features testing --example stress_cluster
./target/profiling/examples/stress_cluster --nodes 3 --mb 50

# Terminal 2 — while it runs, sample for 60 seconds:
sample stress_cluster 60 -file /tmp/hopr-stress.sample.txt
```

The output is a text call tree with inclusive sample counts per frame. Wide frames
at the top of the tree (high inclusive count) are the hot paths — equivalent to wide
frames in a flame graph.

### External perf / cargo-flamegraph (Linux only)

Requires `cargo-flamegraph` (`cargo install flamegraph`) and `perf` (usually
in `linux-tools` / `perf` packages). No extra cargo feature needed.

```bash
cargo flamegraph --profile profiling \
    --features testing \
    --example stress_cluster \
    -- --nodes 5 --mb 100
```

This produces `flamegraph.svg` in the current directory. The flame graph
covers the full process lifetime including cluster bootstrap, which adds
noise; filter to the workload phase by running with `--mb 2` first to
measure bootstrap duration, then use perf's `--delay` flag accordingly.

### Flags reference

| Flag | Default | Description |
|------|---------|-------------|
| `--nodes N` | `5` | Cluster size (2–9). More nodes = more relay combinations but slower bootstrap. |
| `--mb N` | `100` | Megabytes to deliver before stopping. |
| `--routes N` | `4` | Concurrent 1-hop sessions. Each adds a distinct `src → relay → dst` triple. |
| `--seed N` | `42` | RNG seed for route selection. Same seed = identical traffic across runs. |
| `--out PATH` | — | Write flame graph SVG here (`profiling` feature only). |

## Running the smoke test

A `#[ignore]`d integration test validates the harness end-to-end with a
conservative 5 MB volume on 5 nodes:

```bash
cargo nextest run -p hopr-lib \
    --features testing \
    --test 'stress_cluster-size5' \
    -j 1 \
    --run-ignored all \
    -- --nocapture
```

The test asserts delivery ≥ target bytes and at least one non-zero throughput
window. No hard MB/s floor is set — throughput varies with instrumentation
overhead and machine load.

## Interpreting the flame graph

Wide frames near the top of the flame graph are the hot paths. Key frames to
look for:

| Frame | What it covers |
|-------|----------------|
| `hopr_crypto_packet::por` | Proof-of-Relay signing and verification |
| `hopr_crypto_packet::packet` | SPHINX packet encode/decode (Chacha20, poly1305) |
| `hopr_protocol_session` | Session framing, segmentation, ACK state machine |
| `hopr_transport_mixer` | Mixer delay scheduling and packet reordering |
| `tokio::runtime::task` | Async task dispatch overhead |

An unusually wide `tokio::sync::*` frame (mutex, channel, semaphore) suggests
contention. A wide `alloc` frame suggests excessive allocation in a hot path.

## How it works

1. `cluster_fixture(N nodes)` — spawns N per-node tokio runtimes in dedicated
   threads; blocks until full-mesh connectivity and probe warmup are complete
   (can take ~100 s for 3 nodes).
2. Full channel mesh — opens `N*(N-1)` directed channels concurrently via
   `try_join_all`; waits for graph convergence with `wait_for_channel_graph`.
3. Route selection — picks `--routes` distinct `[src, relay, dst]` triples
   using a seeded RNG; same seed = reproducible traffic across profiling runs.
4. Session establishment — opens one long-lived `HoprSession` per triple
   sequentially (each call can take up to ~60 s with retries).
5. Workers — one tokio task per session: `write_all → flush → read_exact`
   in a tight loop. The built-in `EchoServer` on each node echoes bytes back,
   so `read_exact` confirms end-to-end delivery.
6. Sampler — a background task records per-window bytes and MB/s every 500 ms,
   producing the time-series output.
